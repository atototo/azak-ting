# Issue to Docs 워크플로우 검증 체크리스트

이 체크리스트를 사용하여 워크플로우가 올바르게 완료되었는지 검증하세요.

---

## 1. GitHub 이슈 품질

### 이슈 생성

- [ ] GitHub 이슈가 생성되었음
- [ ] 이슈 번호가 정확히 기록됨
- [ ] 이슈 제목이 명확하고 구체적임 (문제/기능이 한눈에 파악됨)

### 이슈 내용

- [ ] 문제 설명이 명확함 (무엇이 잘못되었는지 또는 어떤 기능인지)
- [ ] 재현 방법이 단계별로 작성됨
- [ ] 예상 동작 vs 실제 동작이 명시됨
- [ ] 스크린샷이 첨부됨 (해당되는 경우)
- [ ] 에러 로그가 포함됨 (해당되는 경우)
- [ ] 영향도가 표시됨 (심각도, 범위)
- [ ] 적절한 라벨이 설정됨 (bug, feature, enhancement 등)

---

## 2. 브랜치 및 Git 관리

### 브랜치

- [ ] 브랜치 이름이 규칙을 따름 (`feature/issue-{번호}-{설명}`)
- [ ] 이슈 번호가 브랜치 이름에 포함됨
- [ ] base 브랜치(main/develop)에서 분기됨
- [ ] 최신 상태에서 브랜치 생성됨

### 커밋

- [ ] 커밋 메시지가 명확함
- [ ] 커밋 메시지에 변경 타입 포함 (fix, feat, refactor 등)
- [ ] 커밋 메시지에 이슈 번호 포함 (`Fixes #123`)
- [ ] 불필요한 파일이 커밋에 포함되지 않음 (node_modules, .env 등)
- [ ] 커밋이 논리적 단위로 분리됨 (하나의 거대한 커밋이 아님)

---

## 3. 코드 변경 품질

### 코드 수정

- [ ] 문제가 실제로 해결됨
- [ ] 최소 변경 원칙 준수 (필요한 부분만 수정)
- [ ] 코딩 스타일/컨벤션 준수
- [ ] 타입 안정성 유지 (TypeScript인 경우)
- [ ] 린트 에러 없음
- [ ] 불필요한 리팩토링 없음

### 코드 안정성

- [ ] 기존 기능에 영향 없음 (회귀 없음)
- [ ] 에지 케이스 처리됨
- [ ] 에러 핸들링 적절함
- [ ] 성능 저하 없음

---

## 4. 테스트 완료성

### 자동 테스트

- [ ] 기존 테스트가 모두 통과함
- [ ] 새로운 테스트 추가됨 (필요한 경우)
- [ ] 테스트 커버리지 유지 또는 향상됨
- [ ] 린트/타입 체크 통과

### 수동 테스트

- [ ] 로컬 환경에서 수동 테스트 완료
- [ ] 문제 재현 후 해결 확인
- [ ] 주요 기능 동작 확인
- [ ] 관련 기능들 동작 확인 (회귀 테스트)
- [ ] 다양한 시나리오 테스트 (해당되는 경우)

---

## 5. Pull Request 품질

### PR 생성

- [ ] PR이 생성됨
- [ ] PR 제목이 명확함
- [ ] base 브랜치가 올바름 (main/develop)
- [ ] 이슈와 연결됨 (`Fixes #123`)

### PR 설명

- [ ] 변경 사항 요약이 명확함
- [ ] AS-IS/TO-BE 설명 포함
- [ ] 주요 코드 변경 설명 포함
- [ ] 스크린샷 포함 (Before/After)
- [ ] 테스트 결과 포함
- [ ] 체크리스트 포함

### PR 상태

- [ ] CI/CD 통과 (자동 빌드/테스트)
- [ ] 충돌 없음
- [ ] 리뷰어 지정됨 (필요한 경우)

---

## 6. 변경사항 문서 완성도

### 문서 구조

- [ ] 문서 파일이 생성됨
- [ ] 파일명이 규칙을 따름 (`YYYY-MM-DD-{설명}.md`)
- [ ] 올바른 위치에 저장됨 (`/docs/updates/`)
- [ ] 모든 필수 섹션 포함 (8-10개)
- [ ] 목차 포함

### 문서 내용 - 메타데이터

- [ ] 작업 일자 명시
- [ ] 작업자 명시
- [ ] 관련 이슈 링크
- [ ] PR 링크

### 문서 내용 - 변경 개요

- [ ] 변경 개요가 명확함 (2-3문장으로 핵심 요약)
- [ ] 변경 이유가 설명됨

### 문서 내용 - AS-IS

- [ ] AS-IS 상태가 상세히 설명됨
- [ ] 문제점이 명확히 기술됨
- [ ] AS-IS 스크린샷 포함 (해당되는 경우)
- [ ] 에러 로그 포함 (해당되는 경우)
- [ ] 영향도 테이블 포함

### 문서 내용 - TO-BE

- [ ] TO-BE 상태가 상세히 설명됨
- [ ] 개선사항이 명확히 기술됨
- [ ] TO-BE 스크린샷 포함 (해당되는 경우)
- [ ] Before/After 비교 테이블 포함

### 문서 내용 - 변경 사항 상세

- [ ] 주요 파일 변경 목록 포함
- [ ] 코드 비교 (Before/After) 포함
- [ ] 변경 이유가 각 코드 블록마다 설명됨
- [ ] 코드 스니펫이 적절히 포맷팅됨 (```언어)
- [ ] 아키텍처 변경 설명 포함 (해당되는 경우)

### 문서 내용 - 테스트 결과

- [ ] 자동 테스트 결과 포함
- [ ] 수동 테스트 결과 포함
- [ ] 검증 항목 리스트 포함

### 문서 내용 - 사용 방법

- [ ] 로컬 환경 적용 방법 포함
- [ ] 기능 사용법 설명
- [ ] 예제 코드/명령어 포함 (해당되는 경우)

### 문서 내용 - 참고 사항

- [ ] 변경 요약 테이블 포함
- [ ] 주의 사항 명시
- [ ] 트러블슈팅 가이드 포함
- [ ] 관련 파일 목록 포함
- [ ] 변경 이력 테이블 포함

---

## 7. 스크린샷 품질

### AS-IS 스크린샷

- [ ] AS-IS 스크린샷이 캡처됨
- [ ] 문제 상황이 명확히 보임
- [ ] 스크린샷 해상도가 적절함
- [ ] 올바른 경로에 저장됨 (`/docs/screenshots/`)
- [ ] 파일명이 명확함

### TO-BE 스크린샷

- [ ] TO-BE 스크린샷이 캡처됨
- [ ] 해결된 상태가 명확히 보임
- [ ] AS-IS와 동일한 화면/상황
- [ ] 스크린샷 해상도가 적절함
- [ ] 올바른 경로에 저장됨

### 스크린샷 비교

- [ ] Before/After 비교가 가능함
- [ ] 문서에 적절히 삽입됨
- [ ] 이미지 설명(alt text) 포함

## 8. 전체 프로세스 완료성

### 워크플로우 단계

- [ ] Step 0: 현재 상황 파악 완료
- [ ] Step 1: 문제 체계적 분석 완료 (Analyst)
- [ ] Step 2: AS-IS 캡처 및 문서화 완료 (Tech Writer)
- [ ] Step 3: GitHub 이슈 생성 완료 (PM)
- [ ] Step 4: 브랜치 생성 완료 (Dev)
- [ ] Step 5: 수정 작업 완료 (Dev)
- [ ] Step 6: 테스트 및 TO-BE 캡처 완료 (Dev)
- [ ] Step 7: Pull Request 생성 완료 (SM)
- [ ] Step 8: 변경사항 문서 생성 완료 (Tech Writer)

### 최종 산출물

- [ ] GitHub 이슈 생성됨
- [ ] Git 브랜치 생성 및 푸시됨
- [ ] 커밋 완료됨
- [ ] Pull Request 생성됨
- [ ] 변경사항 문서 생성됨
- [ ] 스크린샷 저장됨
- [ ] 모든 파일이 Git에 커밋됨

### 다음 단계 준비

- [ ] PR 리뷰어 지정됨 (필요한 경우)
- [ ] CI/CD 상태 확인됨
- [ ] 팀원에게 공유 준비됨
- [ ] 배포 계획 수립됨 (필요한 경우)
